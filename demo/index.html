Field notes · a working lakehouse

The table format that remembers every version of your data.

A local Apache Iceberg lakehouse. It ingests the real MovieLens dataset and a synthetic playback-events stream, refines it bronze → silver → gold with dbt-spark, validates every model, and serves it through Trino — storage, catalog, and compute kept as separate, swappable layers.

Most warehouses give you one answer: the current one. Iceberg keeps a full history of snapshots, so you can query the table as it stood last Tuesday, roll a bad load back in seconds, and evolve a schema or partition layout without rewriting a single file.

This lab is a small, honest version of how modern platforms actually run that — on object storage, with two engines reading one set of tables. Scroll to §02 to time-travel through a live snapshot history yourself.

§01

Separation of concerns

storage · catalog · compute

A lakehouse is not one system. It is three independent layers that agree on a contract. Swap any one of them without re-copying your data — that is the leverage on both cost and vendor lock-in.

Storage

MinIO (S3)

Open object storage you control. Parquet data files and Iceberg metadata live here as plain objects — no engine owns them.

s3://warehouse/
Catalog

Iceberg REST

The source of truth for which snapshot is current. Tracks tables, schemas, and the snapshot tree that makes time-travel and rollback possible.

catalog: demo
Compute

Spark + Trino

Spark (with dbt) writes and transforms; Trino reads. Two engines, one set of tables, the same numbers — proof the format is the contract.

Medallion modeling with dbt

dbt-spark

Raw data lands first, then gets conformed, then becomes the business marts analysts query. Each layer is a set of dbt models, tested on every run.

The validation ledger

dbt tests · Great Expectations

§05

Iceberg feature index

demos/
01

Time travel & rollback

demos/01_time_travel.py

Query any past snapshot by id or timestamp, and restore a previous snapshot as the current table in one operation — a recovery that copies nothing.

02

Schema evolution

demos/02_schema_evolution.py

Add, rename, reorder, or drop columns by metadata alone. Old data files stay exactly where they are; readers see the new schema correctly.

03

Hidden partitioning

demos/03_hidden_partitioning.py

Partition by a transform of a column (day of a timestamp) without a separate partition column. Queries prune the right files without users knowing the layout.

04

Compaction & maintenance

demos/04_compaction_maintenance.py

Rewrite many small files into few large ones and expire old snapshots — faster scans and a smaller bill, with the table's data unchanged.

05

Iceberg vs. plain Parquet

demos/05_iceberg_vs_parquet.py

The same data, side by side, with and without the table format — so the cost of atomic commits, snapshots, and pruning is measured, not asserted.

§06

Two engines, one table

Trino

Spark writes the tables; Trino reads the very same files through the same catalog. Note the FOR VERSION AS OF clause — time-travel is first-class SQL.

trino:gold — query at a past snapshot
trino> SELECT title, plays, round(completion_rate, 2) AS compl FROM gold.movie_engagement FOR VERSION AS OF 5 ORDER BY plays DESC LIMIT 4; title | plays | compl --------------------------------+-------+------- Shawshank Redemption, The | 1842 | 0.91 Pulp Fiction | 1730 | 0.86 Forrest Gump | 1689 | 0.88 Matrix, The | 1604 | 0.90 (4 rows · 0.18s · snapshot 5 · scanned 12 files)

Bring up the whole lakehouse with one command.

Docker-composed end to end: MinIO, the Iceberg REST catalog, Spark, and Trino — seeded with real MovieLens data and seven days of synthetic playback events.

$ make up MinIO + Iceberg REST + Spark + Trino
$ make seed && make ingest load bronze
$ make build && make test silver + gold, validated
$ make demo run the 5 Iceberg demos
